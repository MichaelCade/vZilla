FlexPod - Jumbo Frames 

17/06/2014

3 Comments

 
These next few blog posts have really come about because of the conversations with customers over the past 2-3 years and I wanted to be able to dump all of this information in one place…

So I generally work with what is now known as a Flexpod solution, this consists of NetApp Storage Controllers, Cisco UCS Blades or C Class chassis, Cisco Nexus switches and VMware ESXi

The key points I want to discuss are

Jumbo Frames

Network Teaming

Flow Control

Spanning Tree

There are so many other questions relating to similar aspects to the whole stack and networking, however as I am not a profound Networking bod I am not going to pretend I am, there are plenty of Networking gurus out there sharing their views.

What i am saying below is generally my opinion and not always still vendor best practices, these are based on real world scenarios, and if you have any questions please ask away.

Jumbo Frames

Basically when discussing Jumbo Frames we are talking about Ethernet frames that are larger than 1500 bytes, generally and most commonly Jumbo Frames can carry 9000 bytes per payload. The use case for Jumbo Frames is basically to potentially reduce the overheads and CPU cycles, it will also have a positive effect on end to end TCP performance. Although it may have an adverse effect on network latency especially on low bandwidth links.

With the above in mind and the reason why I have included Jumbo Frames on this blog post is because the FlexPod stack utilizes this technology to improve service.

Also as mentioned above Jumbo Frames are only effective if configured from end to end in the stack.

We will start with the Cisco hardware configuration, firstly the UCS will require the Fabric Interconnect to be configured and the vNIC,

Fabric Interconnect MTU size = 9216

vNIC = 9000

the reason for the above setting is because these are the maxiumum MTU size allowed for those devices, the Fabric Interconnect MTU should never be less than the MTU size of the end points, the reason I suggest configuring the Fabric Interconnect to 9216 is because this is the max and avoids the fact that if any padding is added by the OS, Virtual Firewalls or Virtual routers etc.

We then move onto the Nexus 5000 switches,

switch(config)#policy-map type network-qos jumbo

switch(config-pmap-nq)#class type network-qos class-default

switch(config-pmap-c-nq)#mtu 9216switch(config-pmap-c-nq)#exit

switch(config-pmap-nq)#exitswitch(config)#system qos

switch(config-sys-qos)#service-policy type network-qos jumbo

The nexus switch also sets the Jumbo Frame MTU size to 9216 for the same reasons as the Fabric Interconnect.

Our next hardware piece is the virtualisation layer generally in my case is VMware vSphere.

To configure Jumbo Frames on the ESX host it is done on the virtual switch (vswitch) the MTU here is set to 9000, this change is also required on the VMkernel port

We may also have some Nexus 1000v switches included in our configuration, with that in mind these are configured on a port profiles for uplink, the MTU cannot be configured on a vEthernet interface.

show run port-profile UPLINK

the mtu size for the jumbo frames here will be 9000.

Finally we have our NetApp controllers, and simply put here we can configure these using the ifconfig command, this is then generally added to the rc file so that this is consistant upon failures and reboots.

ifconfig e1a-100 192.168.101.105 netmask 255.255.255.0 mtusize 9000partner e1a-100

notice the mtusize 9000 in the command this is all that is required to configure jumbo frames on the netapp controller.

One thing I have not covered in the above is the use of FCOE, it is possible to enable jumbo frames on FCOE interfaces however its not the same payload, the FC payload is 2,112  bytes  and enabling jumbo frames will keep FCoE traffic from being fragmented. On Nexus  5K, we have a pre-defined Class of Service (CoS) to be FCoE, which is by default  has the value 3. Within this CoS, the MTU size it set to 2158.

NetApp Release FAS2500 Series and all Flash Array FAS8080EX

17/06/2014

22 Comments

 
So this morning NetApp released a new family of FAS series controllers, The FAS2500 series provides enterprise class data management in a value priced storage system for all types of businesses.

http://www.netapp.com/UK/products/storage-systems/fas2500/

The new series will replace the FAS2200 series.

The options available for the new series are as follows:

FAS2520
FAS2552
FAS2554

The new systems will ship with Data OnTap 8.2.2 RC1, this will allow for both 7mode and Clustered OnTap.

The new systems all contain disks within the same chassis very much the same as the 2200 series. These controllers where required can be converted into shelves.

NetApp Also released the FAS8080 EX today which is mentioned as being the most powerful storage system that NetApp have produced. This system will replace the FAS6250 and FAS6290.

http://www.netapp.com/uk/system/pdf-reader.aspx?cc=uk&m=ds-3580.pdf&pdfUri=tcm:42-121336

As with the 6250 and 6290 the new 8080 EX will be targeted towards large enterprise businesses.

You will find much more technical detail here at Brian Mitchells Blog post - http://www.ntapgeek.com/2014/06/netapp-debuts-fas8080-ex.html

All in all today has been a great day in terms of new advanced hardware being released by NetApp, not only under the hood the performance gains etc, but also the shiny new bezel
